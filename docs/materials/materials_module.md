**Дата актуализации:** 1 марта 2026 года (внедрена система Global On Focus Refresh)

## Важное замечание о структуре данных

> **Внимание:**
> - Модуль владеет **4 таблицами**: `materials`, `receipts`, `material_aliases`, `kit_components`
> - **Изоляция данных**: Все таблицы содержат `company_id` и изолированы на уровне БД (RLS).
> - **Архитектура**: гибридная — хранение накладных + привязка к сметам через алиасы (N:N)
> - **UI-стандарт**: Единый экран `MaterialScreen` с двумя режимами (М-15, Сводно).
> - **Refresh**: Подключен к глобальной системе обновления при фокусе (TTL 5 минут).
> - **Форматирование**: Все данные форматируются строго через `GtFormatters` (`formatCurrency`, `formatRuDate`, `formatQuantity`).
> - **Критические зависимости**: 
>   - `companies` — владелец всех данных модуля
>   - `estimates` (сметы) — для привязки материалов к каноническим позициям
>   - `contractors` (контрагенты) — для идентификации поставщиков
>   - `work_items` (позиции работ) — для отслеживания использования материалов
> - **Метод списания**: FIFO (First In, First Out) **в единицах материала (накладной)**. Расход из работ пересчитывается в ед. материала через коэффициенты.
> - **Особенность**: Представления `v_materials_with_usage` и `v_materials_grouped_by_estimate` динамически вычисляют использование материалов.

## Детальное описание модуля

Модуль **Материалы** — система учёта материалов на основе накладных с автоматическим списанием по методу FIFO (First In, First Out). 

### Зачем нужен модуль

Модуль решает задачу **сквозного учёта материалов от поставки до использования**:
- Импорт накладных из Excel/CSV в контексте конкретной компании
- **Global On Focus Refresh:** Автоматическое фоновое обновление реестра материалов и сводных данных при возврате в приложение (TTL 5 мин).
- Привязка материалов из накладных к сметным позициям (через алиасы)
- Отслеживание расхода материалов в режиме реального времени
- Расчёт остатков по методу FIFO с учётом договоров и принадлежности к компании
- Контроль дублирования накладных (по номеру, дате, договору и компании)
- **Группировка по смете**: возможность видеть сводный остаток по каноническим позициям (все «Лотки» разных типов в одной строке).

### Ключевые функции

1. **Импорт накладных** — загрузка Excel/CSV, парсинг, сохранение в БД с привязкой к `company_id`
2. **Auto-refresh** — поддержка актуальности остатков при частом переключении между приложениями.
3. **Маппинг материалов** — привязка названий из накладных к сметным позициям через алиасы
4. **Групповой вид (Canonical)** — агрегация всех партий материала под единым сметным наименованием.
5. **FIFO-списание с мультипликаторами** — автоматический пересчет штук в метры (и другие ед. изм.) при списании.
6. **Многоуровневая фильтрация** — по компаниям, договорам, а также индивидуальный поиск по каждой колонке.

### Архитектурные особенности

- **Изоляция Multi-tenancy**: данные разных компаний полностью разделены через RLS.
- **FIFO с пересчетом**: списание происходит в единицах сметы. Если материал пришел в штуках (1 шт = 3 м), система корректно спишет метры из этих штук.
- **Динамический расчет**: использование и остатки вычисляются через SQL Views на лету.
- **Единый интерфейс**: использование `AnimatedSwitcher` для переключения между M-15, Сводной таблицей и Сопоставлением.

## Бизнес-логика

### Переключение режимов (Обновлено 18.01.2026)
В модуле реализовано два режима:
1. **Material M-15**: плоский список всех накладных.
2. **Materials Grouped (Сводно)**: агрегированный вид по смете. Переключается кнопкой «Слои» в AppBar.

### Алгоритм FIFO с коэффициентами (N-to-N)
С 18.01.2026 логика FIFO переведена на расчет в **единицах материала**, что позволяет одному материалу быть привязанным к нескольким работам (N-to-N Mapping).
Если материал в накладной указан в "штуках", а в смете в "метрах" (через алиас задан `multiplier = 3.0`, т.е. 1 шт = 3 м):
1. Расход из работы (в метрах) делится на коэффициент: `material_usage = work_qty / 3.0`.
2. Этот "расход в штуках" суммируется по всем сметным позициям, к которым привязан данный материал в рамках одного договора.
3. Итоговый расход вычитается из прихода (в штуках) по методу FIFO.
4. Это устраняет проблему "общего котла" и позволяет точно отслеживать складской остаток при использовании одного материала в разных узлах сметы.

### Логика привязки к смете (Smart Binding)
При использовании `MaterialFromReceiptsPicker` реализованы следующие правила:
1. **Сортировка**: Уже привязанные к данной позиции материалы всегда отображаются вверху списка.
2. **N-to-N поддержка**: Материалы, привязанные к другим позициям, не блокируются для выбора (статус «ОБЩИЙ»), что позволяет гибко распределять один приход по разным работам.
3. **Нормализация единиц измерения**: Сравнение ед. изм. сметы и накладной происходит после очистки от пробелов, точек и приведения к нижнему регистру. Это исключает ложные предупреждения для «шт», «ШТ.», «шт ».
4. **Статичный UI**: В модальных окнах привязки полностью отключены анимации и сложные Slivers в пользу стабильности `Column` + `ListView`.
5. **Интеллектуальный выбор**: Выбор строки работает как триггер (toggle) по всей области строки (`HitTestBehavior.opaque`).
