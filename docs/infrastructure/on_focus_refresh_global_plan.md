# План внедрения: Global On Focus Refresh (TTL 5 минут)

Статус: Черновик для пошаговой реализации

- Переход к следующему шагу разрешен только после `PASS` текущего шага.
- Если проверка шага не пройдена (`FAIL`) — выполняется доработка и повторная проверка этого же шага.
- Только после повторного `PASS` продолжаем.

## Шаг 2. Coordinator (Riverpod)

- [ ] Создать `AppFocusRefreshCoordinator` в `core`.
- [ ] Реализовать API:
  - `registerTarget(...)`
  - `unregisterTarget(...)`
  - `markTargetVisible(id, bool)`
  - `handleAppResumed()`
- [ ] В `handleAppResumed()` добавить:
  - lock от параллельного запуска (`isRefreshing`)
  - проверку TTL = 5 минут
  - запуск только разрешенных targets
  - обновление `lastRunByTargetUtc`

### Проверка шага 2

- [ ] Target с интервалом < 5 минут не запускается.
- [ ] Target с интервалом >= 5 минут запускается.
- [ ] Повторный `onResume` во время refresh не создает второй цикл.

### Решение по шагу 2

- [ ] `PASS` / [ ] `FAIL`
- [ ] Если `FAIL`: доработка шага 2 выполнена, повторная проверка пройдена.

## Шаг 3. Глобальный lifecycle-listener

- [ ] Подключить один `AppLifecycleListener` в корневом слое приложения.
- [ ] На `onResume` вызывать `coordinator.handleAppResumed()`.
- [ ] Проверить, что listener инициализируется один раз.

### Проверка шага 3

- [ ] `onResume` доходит до coordinator.
- [ ] Нет дублей вызовов из нескольких listener.

### Решение по шагу 3

- [ ] `PASS` / [ ] `FAIL`
- [ ] Если `FAIL`: доработка шага 3 выполнена, повторная проверка пройдена.

## Шаг 4. Пилотная интеграция (estimates)

- [ ] Зарегистрировать target `estimates` с TTL 5 минут.
- [ ] В callback сделать точечный `ref.invalidate(...)` только нужных provider.
- [ ] При закрытии экрана/фичи сделать `unregisterTarget`.

### Проверка шага 4

- [ ] Возврат в приложение < 5 минут: refresh не выполняется.
- [ ] Возврат в приложение >= 5 минут: refresh выполняется 1 раз.
- [ ] Нет сброса выбора/фильтров/скролла.

### Решение по шагу 4
